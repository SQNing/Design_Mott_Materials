## Per-Feature Details

### Sunny / Julia Phase 1 LSWT environment migration
- **Role:** Downstream user validating that FeI2 `V2b` can now reach the real Sunny LSWT backend instead of dying in environment setup.
- **Use Case:** Provision Julia 1.12.6, instantiate `.julia-env-v07`, verify `using JSON3, Sunny`, probe the LSWT driver on a minimal spin-frame model, then rerun the real FeI2 `V2b` downstream smoke with `DESIGN_MOTT_JULIA_CMD=/data/work/zhli/soft/julia-1.12.6/bin/julia`.
- **What they tried:** Added the new repo-local LSWT environment, rewired `run_sunny_lswt.jl` to `.julia-env-v07`, added explicit Julia command resolution in `linear_spin_wave_driver.py`, instantiated the environment against `scripts/.julia-depot`, and reran both a direct LSWT probe and the real FeI2 `run_document_reader_pipeline(...)` smoke.
- **Discoverability:** The new environment contract is explicit in both code and docs; the driver-level `DESIGN_MOTT_JULIA_CMD` override makes the non-default Julia 1.12.6 path discoverable without hardcoding it into the repo.
- **Setup:** Julia 1.12.6 was installed under `/data/work/zhli/soft/julia-1.12.6`, and `.julia-env-v07/Manifest.toml` resolved `Sunny v0.7.8` plus `JSON3 v1.14.3`.
- **Functionality:** The direct LSWT probe and the real FeI2 smoke both moved beyond environment bootstrapping. The FeI2 rerun preserved `lswt=ready`, `gswt=blocked`, and `thermodynamics=review`, and the `lswt` stage now fails with `backend-execution-failed: Not an energy-minimum; wavevector q = [0.0, 0.0, 0.0] unstable.` rather than a missing-package error.
- **Expected vs Actual Outcome:** Phase 1 succeeded at its environment goal. Full LSWT success is still blocked, but the blocker is now a real Sunny/backend stability issue rather than an inconsistent Julia/Sunny environment.
- **Blocked steps:** FeI2 LSWT still needs a physically stable classical reference or a more backend-compatible LSWT setup to eliminate the new instability error.
- **Friction points:** The first rerun attempt accidentally omitted `DESIGN_MOTT_JULIA_CMD`, which reproduced a legacy `missing-json3-package` symptom; once the env var was correctly injected, the new environment behaved as intended.
- **Doc suggestions:** Keep the new environment reference, and add a short note near FeI2 or LSWT workflow docs that Phase 1 removes environment blockers but does not guarantee the supplied classical state is a Sunny energy minimum.

## Expected vs Actual Outcome

The Phase 1 migration achieved its intended technical goal:

- Julia 1.12.6 is installed and usable
- `.julia-env-v07` instantiates successfully
- `Sunny v0.7.8` and `JSON3 v1.14.3` load from the repo-local project/depot
- the LSWT backend now reaches real Sunny execution

The remaining mismatch is no longer environmental. In the final FeI2 rerun, `lswt` fails with:

- code: `backend-execution-failed`
- message: `Not an energy-minimum; wavevector q = [0.0, 0.0, 0.0] unstable.`

That is a narrower and more truthful downstream issue than the earlier `missing-sunny-package` / `missing-json3-package` failures.

## Issues Found

- The Phase 1 environment migration is successful, but the current FeI2 classical reference still appears unstable for Sunny LSWT, so `lswt` remains `error` even though the environment is now correct.

## Suggestions

- Treat Phase 1 as complete for the LSWT environment baseline: the repo can now reproduce real Sunny runtime behavior.
- Use the new narrowed error to decide the next scoped step: either improve FeI2 LSWT stability inputs, or defer that physics/backend follow-up while moving other Sunny-family work into Phase 2.
