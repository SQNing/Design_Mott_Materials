## Per-Feature Details

### FeI2 `V2b` downstream chaining
- **Role:** Downstream user validating the first productized FeI2 bridge-to-downstream workflow.
- **Use Case:** Start from a real FeI2 normalized document, reuse the landed `V2a` shell-expanded bridge, solve the classical state, and inspect the new `V2b` route and execution artifacts.
- **What they tried:** Ran `run_document_reader_pipeline(...)` on `FeI2/input.tex` with a previously successful real `agent_normalized_document.json`, `selected_model_candidate="effective"`, `selected_local_bond_family="2a'"`, `selected_coordinate_convention="global_crystallographic"`, `emit_spin_only_solver_payload=True`, `run_spin_only_classical_solver=True`, and `run_downstream_stages=True`.
- **Discoverability:** The new route/result artifact layout is understandable once the run finishes; the main hidden requirement is that LSWT execution still depends on a working Sunny.jl environment.
- **Setup:** No extra repository setup was needed, but the active Julia environment did not contain Sunny.jl.
- **Functionality:** The bridge and classical stages succeeded. `downstream_routes.json` shows `lswt=ready`, `gswt=blocked` with reason `requires-local-ray-cpn-state`, and `thermodynamics=review` with reason `requires-caller-confirmed-support`. `lswt` did execute and produced a stage result, but the result status was `error` because Sunny.jl was unavailable.
- **Expected vs Actual Outcome:** The route/result/summarization contract worked as intended. The only mismatch was environmental: the auto-executed `lswt` stage failed at runtime with `missing-sunny-package`.
- **Blocked steps:** Full LSWT success is blocked until the active Julia environment provides Sunny.jl.
- **Friction points:** `recommended_backend="linear_spin_wave"` does not immediately tell a user that the underlying implementation still requires Sunny.jl.
- **Doc suggestions:** Note in the downstream-stage docs or FeI2 workflow docs that the current LSWT backend still depends on Sunny.jl being installed.

## Expected vs Actual Outcome

The main `V2b` contract was validated. The pipeline emitted all three downstream artifact files and preserved stage-level semantics cleanly:

- `lswt` was `ready` and attempted execution
- `gswt` was `blocked` and was not executed
- `thermodynamics` was `review` and was not auto-executed

The actual execution result was `partial` rather than fully successful because `lswt` failed with:

- code: `missing-sunny-package`
- message: `Sunny.jl is not available in the active Julia environment`

## Issues Found

- The first real FeI2 `V2b` smoke still depends on an external Sunny.jl installation for LSWT execution, so a `ready` LSWT route can still terminate in an environment-level runtime error.

## Suggestions

- Keep the current route/result artifact design; it made the failure diagnosable without source inspection.
- Surface the Sunny.jl dependency more explicitly near LSWT execution docs or downstream summaries.
